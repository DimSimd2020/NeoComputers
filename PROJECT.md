\# NeoComputers

> \*\*The Next Generation of Computing in Minecraft\*\*

> 

> Полноценная эмуляция Linux-окружения внутри Minecraft 1.21 на базе NeoForge с упором на экстремальную производительность, 3D-визуал и 99% кастомизацию.



\## 1. Vision \& Core Concept

NeoComputers — это духовный наследник OpenComputers, переосмысленный для современных версий игры. Вместо ограниченных скриптовых языков (Lua), мод предоставляет виртуализированное аппаратное обеспечение, способное запускать реальные Linux-дистрибутивы (архитектура RISC-V или x86\_64).



\## 2. Technical Stack

\* \*\*Game Engine:\*\* Minecraft 1.21

\* \*\*Mod Loader:\*\* NeoForge

\* \*\*Language:\*\* Java 21 (Logic \& Integration)

\* \*\*Emulation Core:\*\* C++ (Native Backend)

\* \*\*Interoperability:\*\* Project Panama (FFM API)

\* \*\*Graphics:\*\* Custom OpenGL Shaders / Dynamic Texture Atlas

\* \*\*Models:\*\* Blockbench (High-poly 3D models for peripherals)



\## 3. Non-Goals (Scope Control)

Критически важные ограничения первой итерации для предотвращения "раздувания" проекта:

\* \*\*No full GPU passthrough:\*\* Виртуальная видеокарта работает через простой фреймбуфер. Никакого прямого доступа к аппаратным ресурсам хоста.

\* \*\*No complete PCI ecosystem:\*\* Только базовый набор виртуальных устройств (шина, память, диск, терминал).

\* \*\*No full Wayland/X11 compatibility in MVP:\*\* Фокус на консольном взаимодействии (CLI) и кастомных GUI-приложениях. Полноценные оконные менеджеры — вне текущих планов.

\* \*\*No SMP in early builds:\*\* Эмуляция будет одноядерной для упрощения синхронизации состояний.

\* \*\*No internet bridge to host OS initially:\*\* Полная изоляция от сети хоста во избежание эксплойтов и спама.



\## 4. Performance Budget

Целевые показатели для сохранения стабильности игрового процесса (TPS):

\* \*\*Max ms per tick (Game Thread):\*\* < 0.5ms на обработку всей логики Neo-Core.

\* \*\*Target VM Update Frequency:\*\* 60Hz (синхронизация состояния VM вынесена в асинхронный поток).

\* \*\*Screen Refresh Rate Cap:\*\* 20 FPS (для снижения нагрузки на рендер и пакетную передачу).

\* \*\*Entity Limit:\*\* Не более 8 активных компьютеров на чанк / 32 на сервер в базовой конфигурации.



\## 5. Security Model

Изоляция — приоритет №1:

\* \*\*VM Isolation:\*\* Виртуальная машина полностью изолирована от адресного пространства хоста.

\* \*\*Filesystem Sandbox:\*\* Никакого прямого доступа к файловой системе сервера. Образы дисков хранятся в зашифрованных/бинарных контейнерах.

\* \*\*Code Integrity:\*\* Запрет на выполнение произвольного нативного кода со стороны пользовательских образов дисков.

\* \*\*Device Whitelist:\*\* Только строго определенные виртуальные устройства экспонируются внутри Linux-окружения.



\## 6. Save/Load \& Persistence Model

\* \*\*Disk Storage:\*\* Диски хранятся как отдельные `.img` файлы в папке мира (`/world/neocomputers/disks/`).

\* \*\*State Snapshots:\*\* Состояние оперативной памяти (RAM) дампится при выгрузке чанка или остановке сервера (опционально, зависит от конфига).

\* \*\*Chunk Unload Behaviour:\*\* При выгрузке чанка VM переходит в режим паузы (SUSPEND) и сохраняет состояние регистров.

\* \*\*Server Restart:\*\* При рестарте сервера компьютеры пытаются восстановиться из последнего снапшота (Cold Boot / Warm Boot настройки).



\## 7. Key Features (The "99% Customization" Rule)

\* \*\*Extreme Visuals:\*\* Полноценные 3D-модели (корпуса, клавиатуры, мыши) с RGB-подсветкой (HEX-поддержка).

\* \*\*Modular Hardware:\*\* Сменные процессоры, плашки памяти, видеокарты и визуально вращающиеся кулеры.

\* \*\*Cable Management:\*\* 3D-провода формата "мультипарт" с кастомизацией оплетки и материалов.

\* \*\*Seamless Monitors:\*\* Мультиблочные дисплеи без рамок.


\## 8. Repository Layout

Репозиторий разделяется на жесткие зоны владения:

\* \*\*/panama-bridge/\*\* - единственная общая зона. Здесь живут стабильные контракты, ABI/API-описания и FFM-совместимые структуры данных. Никакой бизнес-логики и никакой реализации эмулятора.

\* \*\*/neocore-native/\*\* - домен C++-ядра и эмулятора. Здесь допускаются только нативный backend, низкоуровневые типы и реализация контрактов из \*/panama-bridge/\*.

\* \*\*/neocore-forge/\*\* - домен Java и NeoForge. Здесь живут игровые интеграции, UI, блоки, предметы, сети, события и вызовы в \*/panama-bridge/\*.


\## 9. Multi-Agent Workflow

Стандартный порядок работы:

1. Сначала фиксируется контракт в \*/panama-bridge/\*.
2. После фиксации контракта задачи раздаются по изолированным веткам:
   \* \`agent/5.3-cpp-core\`
   \* \`agent/5.4-forge-ui\`
3. GPT-5.3 работает только в \*/neocore-native/\* и получает только нативный контракт и узкое ТЗ.
4. GPT-5.4 работает только в \*/neocore-forge/\* и получает только Java/NeoForge-контекст и контрактные заголовки.
5. Стыковка изменений делается человеком или отдельной сессией, которая видит только готовый diff, но не пытается переписывать чужую область владения.
